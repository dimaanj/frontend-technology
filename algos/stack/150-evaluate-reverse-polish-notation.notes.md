# 150. Evaluate Reverse Polish Notation — итоги

## Сложность

- **По времени:** O(n), где n — число токенов. Один проход по массиву, каждая операция со стеком O(1).
- **По памяти:** O(n). В худшем случае все токены — операнды и попадают в стек (например, много чисел подряд и операторы в конце); для валидного RPN глубина стека порядка O(n).

## Альтернативы

- **Стек (текущее решение):** классический подход для RPN — один проход слева направо, операнды в стек, при операторе достаём два верхних, считаем, кладём результат. Просто и эффективно.
- **Рекурсия (разбор с конца):** последний токен — корень выражения. Если это число — возвращаем его. Если оператор — рекурсивно вычисляем правый операнд (токены перед последним), затем левый; применяем оператор. Индекс идёт с конца к началу. Сложность та же O(n) по времени и O(n) по памяти (глубина рекурсии ≈ количеству операторов). Реализация: `evalRPNRecursive` в том же файле.
- **Map операторов:** вместо `switch` можно хранить объект с операторами и функциями `(a, b) => a + b` и т.д. — чуть гибче при добавлении операций, по сути то же решение.

## Оптимизация

- Текущее решение уже оптимально по асимптотике. Можно заменить `switch` на объект с функциями — только для читаемости/расширяемости.
- `Number(token)` для операндов — нормальный выбор; парсинг один раз на токен.

## Краевые случаи (покрыты тестами)

- Один токен (одно число) — возврат этого числа.
- Отрицательные числа в токенах (`"-5"`, `"-11"`).
- Деление с усечением к нулю: `Math.trunc` — 7/2 → 3, -7/2 → -3, 7/-2 → -3.
- Все четыре оператора: +, -, *, /.
- Сложное выражение из условия (Example 3) — порядок операций и вложенность.
