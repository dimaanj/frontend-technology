# QuickSort — саммари

## Идея

**Divide and conquer:** выбираем **pivot**, делаем **partition** (все меньшие — слева, большие или равные — справа), pivot оказывается на своей итоговой позиции, затем рекурсивно сортируем левый и правый отрезки.

## Зачем переносим pivot в конец (swap arr[mid] и arr[hi])

Перед циклом partition мы ставим выбранный pivot (например, из середины) в **конец** отрезка — в `arr[hi]`.

- **Удобство одного цикла:** в цикле проходим по индексам `j` от `lo` до `hi - 1`, не трогая `hi`. Pivot не участвует в сравнениях и перестановках — мы с ним не «сталкиваемся» в цикле.
- **Один финальный swap:** после цикла граница «меньших» — индекс `i`. Мы один раз делаем `swap(arr[i], arr[hi])` и pivot оказывается на своей итоговой позиции `i`.

Если pivot не переносить в конец, при проходе `j` мы будем натыкаться на сам pivot и нужна отдельная обработка (пропуск или особые условия). Перенос в конец даёт одну простую схему: цикл по `j < hi`, затем один обмен с `arr[hi]` и возврат `i`.

## Алгоритм

1. **Базовый случай:** если `lo >= hi` — выходим.
2. **Partition:**
   - Выбрать pivot (часто `mid = (lo + hi) >> 1`), swap с `arr[hi]`, `pivotValue = arr[hi]`.
   - Два указателя: `i = lo` (граница «меньших»), `j` от `lo` до `hi - 1`. Если `arr[j] < pivotValue` — swap `arr[i]` и `arr[j]`, `i++`.
   - В конце: `swap(arr[i], arr[hi])`, вернуть `i`.
3. **Рекурсия:** `quickSort(arr, lo, pivotIndex - 1)` и `quickSort(arr, pivotIndex + 1, hi)`.

## Сложность

|          | Время      | Память   |
|----------|------------|----------|
| Среднее  | O(n log n) | O(log n) |
| Худшее   | O(n²)      | O(n)     |
| Лучшее   | O(n log n) | O(log n) |

- В среднем разбиение близко к пополам → высота дерева O(log n), на каждом уровне O(n) → O(n log n).
- В худшем (например, уже отсортированный массив и pivot всегда минимум/максимум) — разбиение 1 и n−1 → O(n²). Выбор pivot из середины (или рандом) уменьшает вероятность.
- Память: только стек рекурсии, доп. массивов нет → O(log n) в среднем, O(n) в худшем.

## Особенности

- **Стабильность:** нет в типичной in-place реализации (swap меняет порядок равных).
- **In-place:** да, только обмены в исходном массиве.
- **Хорош для:** среднего случая, когда важна малая доп. память и не нужна стабильность.

## Edge cases

- Пустой массив, один элемент — `lo >= hi`, выход.
- Дубликаты — partition должен ставить «равные» в одну сторону (например, все `>= pivot` справа); корректность сохраняется.
