# 146. LRU Cache — итоги разбора

[LRU Visualizer](https://lrucache.netlify.app/)

## Решение

Использованы **Map** (key → Node) и **двусвязный список** с фиктивными head/tail. MRU у головы, LRU у хвоста.

- **get(key):** если ключ есть — remove(node), addToHead(node), return value; иначе -1.
- **put(key, value):** если ключ есть — обновить value, remove, addToHead; иначе — новый Node, map.set, addToHead, при `map.size > capacity` удалить tail.prev из списка и из map.

## Сложность

- **Время:** O(1) в среднем для get и put (доступ в Map O(1), remove и addToHead — перекидывание указателей O(1)).
- **Память:** O(capacity) — не более capacity узлов в map и в списке, плюс два dummy-узла.

## Альтернативы

- **Map + двусвязный список** (текущий вариант) — стандартный способ достичь O(1) для get/put и eviction.
- **Только Map:** нельзя за O(1) определить и удалить LRU (нет порядка использования).
- **Ordered structure (например, объект с порядком ключей):** в JS порядок ключей не даёт LRU без дополнительной структуры; eviction по «последнему использованному» потребовал бы пересчёта.
- **Два Map (key→value, key→timestamp) + перебор по eviction:** eviction не O(1).

Вывод: связка Map + двусвязный список — подходящий вариант для ограничения O(1).

## Оптимизация

- Текущая реализация уже даёт O(1) по времени; узкое место — только константы (аллокации Node при put нового ключа).
- Можно не создавать новый Node при put существующего ключа — только обновить value и перекинуть узел (так и сделано).

## Краевые случаи

- **capacity 1:** при новом put вытесняется единственный ключ; put того же ключа — только обновление, без eviction.
- **key 0 / value 0:** по условию допустимы (0 <= key, 0 <= value); учтены в тестах.
- **get отсутствующего ключа:** возврат -1.
- **Пустой кэш:** get любого ключа даёт -1.
- **Порядок eviction:** вытесняется всегда tail.prev (наименее недавно использованный); get и put обновляют порядок через remove + addToHead.
