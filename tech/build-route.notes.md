### Сложность

- **Время**: O(n).  
  Несколько линейных проходов по массиву билетов: поиск всех `to`, поиск стартового города по `from`, построение словаря `from → билет` и один проход по цепочке маршрута.
- **Память**: O(n).  
  Используются две структуры: `Map` с городами назначения и `Map` `from → билет`, каждая в худшем случае хранит все билеты/города.

### Альтернативы

- **Линейный подход через граф без дополнительных структур**:  
  Можно сначала найти стартовый город через множества `from` и `to`, а затем в цикле каждый раз искать следующий билет линейным поиском по массиву. Это даёт худшую сложность O(n²), но избавляет от словарей/карт.
- **Использование двух `Set` вместо `Map` на этапе поиска старта**:  
  Хранить отдельно множества всех `from` и всех `to`, затем взять разность множеств, чтобы найти старт. Логика та же, что и в решении, но акцент на множества, а не на карту городов назначения.

### Оптимизация

- **Текущее решение уже оптимально по асимптотике**:  
  O(n) по времени и O(n) по памяти — лучше, чем наивный O(n²) с двойным циклом.
- **Микрооптимизации**:  
  Можно объединить сбор `toCities` и `fromTo` в один проход по массиву билетов, но это меняет только константу, а не асимптотику.

### Краевые случаи

- Пустой массив билетов: нужно явно определиться, что возвращать (обычно `[]`), хотя по условию может не встречаться.
- Маршрут из одного билета: корректно обрабатывается тем же алгоритмом (старт — `from` единственного билета, цикл по `Map` выполнится один раз).
- Повторяющиеся города как промежуточные точки: по условию петель и повторов нет, поэтому решение не проверяет и не обрабатывает случаи с циклами или раздвоениями маршрута.
