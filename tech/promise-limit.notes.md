# promise-limit — итоги обсуждения

## Задача

Реализовать функцию `promiseLimit(tasks, limit)`:
- `tasks` — массив фабрик `() => Promise`
- `limit` — максимальное число одновременно выполняющихся промисов
- Результаты возвращаются в **исходном порядке** tasks
- Если задача падает — ошибка попадает в соответствующую позицию результата, остальные задачи **продолжают выполняться**

---

## Решение — подход «скользящее окно» (worker pool)

```js
async function promiseLimit(tasks, limit) {
  const results = [];
  const executing = [];
  for (let i = 0; i < tasks.length; i++) {
    const task = tasks[i]()
      .then((result) => {
        results[i] = result;
        executing.splice(executing.indexOf(task), 1);
        return result;
      })
      .catch((error) => {
        results[i] = { error };
        executing.splice(executing.indexOf(task), 1);
        // throw убран — иначе Promise.race/all бросил бы исключение наружу
      });

    executing.push(task);

    if (executing.length >= limit) {
      await Promise.race(executing);
    }
  }
  await Promise.all(executing);
  return results;
}
```

### Как работает

1. Запускаем задачу `tasks[i]()`, получаем промис `task`.
2. В `.then` / `.catch` сохраняем результат/ошибку в `results[i]` по оригинальному индексу, затем удаляем себя из `executing` через `splice`.
3. Добавляем `task` в `executing`.
4. Если `executing.length >= limit` — встаём на паузу через `Promise.race`: ждём, пока **любой** из текущих промисов завершится и освободит слот.
5. После цикла дожидаемся хвостовых задач через `Promise.all(executing)`.

---

## Сложность

| | |
|---|---|
| **Время** | O(N) — каждая задача запускается и завершается ровно один раз |
| **Память** | O(limit) — в `executing` одновременно не более `limit` промисов; `results` — O(N) |

---

## Баг и его исправление

**Проблема:** исходный `.catch` содержал `throw error`. Это оставляло промис в состоянии rejected, и `Promise.race` / `Promise.all` выбрасывали исключение наружу, прерывая весь цикл — нарушалось требование «продолжать остальные задачи при ошибке».

**Фикс:** убрать `throw error`. Ошибка уже записана в `results[i]`, промис тихо resolve-ится, цикл не прерывается.

## Второе решение — воркеры (`promiseLimit1`)

### Идея

Запустить ровно `Math.min(limit, tasks.length)` независимых воркеров. Каждый воркер — **самоподдерживающаяся рекурсивная цепочка**: выполнил задачу → сразу тянет следующую из общей очереди через `nextIndex++`.

### Ключевые моменты

**1. Общая очередь через `nextIndex`**
Единственная переменная `nextIndex` разделяется между всеми воркерами. Гонки нет — JS однопоточный, `nextIndex++` атомарен в рамках event loop.

**2. `run()` возвращает промис всей цепочки**
`run` рекурсивно вызывает себя через `.then`, поэтому промис разрешится только когда воркер **исчерпает все свои задачи** (рекурсия остановится при `nextIndex >= tasks.length`).

```
runners[0] = run(A) → A завершился → run(C) → C завершился → run(E) → конец
runners[1] = run(B) → B завершился → run(D) → D завершился → конец
```

**3. Зачем `Promise.all(runners)`**
Без него `res` вернётся пустым — воркеры ещё работают. `Promise.all` — единственная точка ожидания: ждёт, пока **каждый воркер** дойдёт до конца цепочки. В первом решении ожидание распределено по циклу (`await Promise.race`), здесь — вынесено в одно место.

**4. `Math.min(limit, tasks.length)` при запуске**
Защита от случая, когда задач меньше лимита — не создаём лишних воркеров.

### Схема работы

```
tasks: [A, B, C, D, E],  limit: 2

Запуск: воркер 1 → A (nextIndex=2), воркер 2 → B (nextIndex=2... нет: nextIndex=1→2)
A готов  → воркер 1 берёт C (nextIndex=3)
B готов  → воркер 2 берёт D (nextIndex=4)
C готов  → воркер 1 берёт E (nextIndex=5)
D готов  → воркер 2: nextIndex >= tasks.length → стоп
E готов  → воркер 1: стоп
Promise.all завершён → return res
```

---

## Сравнение подходов

| | `promiseLimit` (race) | `promiseLimit1` (workers) |
|---|---|---|
| Управление слотами | Явный `executing[]` + `splice` | Неявно через рекурсию |
| Синхронизация | `Promise.race` в цикле | `Promise.all` в конце |
| Точка ожидания | Распределена по циклу | Одна — в конце |
| Читаемость | Сложнее (замыкание на `task`) | Проще, нет `splice` |
| Нюанс | `indexOf(task)` — хрупко | `nextIndex++` — безопасно |

---

## Паттерн

Оба решения реализуют **worker pool / concurrency limiter** — аналог классического sliding window, где «скольжение» происходит не по индексу, а по событию завершения промиса (выбывает быстрейший, не левый).
