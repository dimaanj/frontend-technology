## Ограничение числа параллельных промисов (подход с воркерами)

### Формулировка задачи

Нужно реализовать `promiseLimit(tasks, limit)`, где:
- **`tasks`** — массив функций без аргументов, каждая возвращает `Promise`;
- **`limit`** — максимум промисов, которые могут выполняться одновременно;
- функция возвращает **один промис**, который резолвится **массивом результатов в исходном порядке**;
- если какая‑то задача падает, остальные должны **продолжить выполняться**, а на позиции упавшей задачи в результирующем массиве должен быть её результат (ошибка или её представление).

Ключевые требования:
- не запускать больше чем `limit` задач одновременно;
- при завершении любой задачи запускать следующую из очереди (если она есть);
- сохранить соответствие «индекс задачи → индекс в результирующем массиве»;
- не блокировать event loop (никаких «бесконечных» синхронных циклов).

---

### Почему нельзя просто сделать `Promise.all(tasks.map(fn => fn()))`

Если написать:

```js
await Promise.all(tasks.map(fn => fn()));
```

то:
- **все задачи** стартуют сразу — никакого лимита по параллельности нет;
- порядок результатов при этом сохранится (по индексу), но ограничение `limit` нарушено.

Нам нужна обёртка, которая:
- управляет тем, **когда именно** какие задачи запускаются;
- но при этом даёт наружу один общий `Promise`, как `Promise.all`.

---

### Идея подхода: фиксированное число «воркеров»

Вместо того чтобы запускать все задачи сразу, мы запускаем **ровно `limit` «воркеров»**:
- каждый воркер — это async‑функция, которая:
  - берёт текущий индекс из общей очереди (например, из переменной `nextIndex`);
  - запускает `tasks[index]()` и `await`‑ит её;
  - по завершении **кладёт результат в правильную позицию** массива `results[index]`;
  - затем берёт **следующий** индекс и повторяет, пока задачи не закончатся.

При этом:
- одновременно выполняется не больше, чем `limit` «висящих» `await tasks[i]()` — это и есть наш лимит;
- как только одна задача завершилась, воркер тут же запускает следующую — за счёт этого поддерживается «конвейер»;
- порядок результатов сохраняется, потому что каждый результат/ошибка всегда пишется в `results[index]`, где `index` — исходный индекс задачи;
- `Promise.all` запускается **над воркерами**, а не над всеми задачами сразу.

Важно понимать, что **«слот» — это не один промис, а целая цепочка промисов**. Для каждого слота мы создаём один начальный промис (например, `run(startIndex)`), а когда задача в этом слоте завершается, внутри `then`/`catch` мы можем вернуть `run(nextIndex)` — тем самым **продолжая цепочку в том же слоте уже с новой задачей**. Снаружи мы по‑прежнему видим один промис на слот, но внутри него по цепочке выполняются несколько задач по очереди.

---

### Структура решения (логика, без «готового кода»)

1. **Подготовка состояния**
   - `results = new Array(tasks.length);` — в этот массив будем складывать результаты.
   - `let nextIndex = 0;` — индекс следующей ещё не запущенной задачи.
   - `const total = tasks.length;`

2. **Функция воркера**
   - Внутри цикла:
     - если `nextIndex >= total`, воркер завершает работу (возвращает управление).
     - берёт локально `current = nextIndex; nextIndex++`;
     - запускает `tasks[current]()` и `await`‑ит:
       - при `resolve` пишет `results[current] = value`;
       - при `reject` пишет `results[current] = error` (или оборачивает ошибку как‑то иначе);
     - после этого цикл повторяется — воркер берёт следующий индекс, если он ещё есть.

3. **Запуск воркеров**
   - Создаём массив промисов воркеров:
     - максимум `limit`, но не больше `tasks.length`.
   - Для каждого от 0 до `limit - 1`:
     - стартуем воркер, который будет обрабатывать цепочку задач (0, 1, 2, ... и т.д. в зависимости от общего `nextIndex`).

4. **Ожидание завершения**
   - `await Promise.all(runners);` — ждём завершения всех воркеров.
   - Возвращаем `results`.

### Каркас решения (скелет из кода)

Ниже — скелет из текущей реализации как демонстрация каркаса. В нём уже есть: массив результатов, очередь воркеров, функция `run(fn, index)` и цикл запуска первых `limit` задач. Нужно дописать: запись результата/ошибки в `res[index]`, рекурсивный/циклический вызов следующей задачи после завершения текущей и корректную инициализацию `res` (размер = `tasks.length`).

```js
const res = [];

let curIndex = 0;
const runners = [];

function run(fn, index) {

  return fn()
    .then((data) => {

    })
    .catch(() => {});
}

for (let i = 0; i < limit; i++) {
  runners.push(run(tasks[i]), i);
}

await Promise.all(runners);

return res;
```

**Что доработать в скелете:**
- В `run`: записывать `data` в `res[index]` при успехе и ошибку в `res[index]` при `catch`; после завершения (в `then`/`catch` или `finally`) проверять, есть ли следующая задача (`curIndex < tasks.length`), и если да — вызывать `run(tasks[curIndex], curIndex)` с последующим `curIndex++`, возвращая этот промис (чтобы воркер «продолжил цепочку»).
- Массив `res`: инициализировать размером `tasks.length` (например, `new Array(tasks.length)`), чтобы порядок результатов совпадал с порядком задач.
- Цикл старта: передавать в `run` правильную пару `(tasks[i], i)` и не допускать выхода за границы массива (если `tasks.length < limit`, запускать только `tasks.length` воркеров).

---

Важно: хотя `Promise.all(runners)` выглядит как «запустить всё сразу», на самом деле:
- мы параллельно запускаем только **`limit` воркеров**;
- каждый воркер в каждый момент времени **ждёт только одну задачу** (`await tasks[i]()`), и только после её завершения запускает следующую;
- поэтому **в каждый момент времени** активных задач не больше, чем `limit`.

---

### Сохранение исходного порядка результатов

Ключевой приём: **информация об индексе задачи не теряется**.

- У нас есть массив `tasks`, где задача с индексом `i` — это `tasks[i]`.
- Когда воркер забирает задачу:
  - он всегда знает её исходный индекс `current`;
  - результат/ошибку пишет ровно в `results[current]`.

Из‑за этого:
- не важно, какая задача завершится первой — 0‑я, 3‑я, 10‑я;
- результирующий массив всё равно будет в исходном порядке:
  - `results[0]` — от `tasks[0]`,
  - `results[1]` — от `tasks[1]`,
  - и т.д.

Таким образом мы **развязываем**:
- порядок старта/завершения задач (управляется ограничением по параллельности);
- и порядок итогового массива (жёстко привязан к индексам в `tasks`).

---

### Не блокируем event loop

Важный анти‑пример — «горячий» цикл вида:

```js
while (есть ещё незапущенные задачи) {
  // что‑то делаем, запускаем промисы
}
```

Если внутри нет `await` или других разрывов (типа `setTimeout`), такой цикл:
- крутится синхронно,
- **не даёт управлению вернуться в event loop**,
- из‑за этого колбэки промисов (`then`/`catch`) не попадают на выполнение,
- можно легко получить «зависание» и повторный запуск одних и тех же задач.

В воркер‑подходе:
- внутри воркера мы используем `await tasks[current]();`
- `await` **приостанавливает** текущую async‑функцию и отдаёт управление обратно event loop;
- другие воркеры, таймеры, обработчики промисов свободно выполняются;
- после завершения задачи воркер продолжает работу и берёт следующую.

То есть:
- мы соблюдаем ограничение по параллельности;
- не блокируем event loop;
- и корректно обрабатываем завершение/ошибки.

---

### Сложность

- **По времени**:
  - суммарно выполняются все задачи, поэтому нижняя граница — сумма их времён;
  - ограничение по параллельности может растянуть общее время по сравнению с «запустить всё сразу»;
  - накладные расходы решения — \(O(n)\) по числу задач (индексация, запись результатов, управление очередью).

- **По памяти**:
  - массив `results` размера `n` → \(O(n)\);
  - несколько служебных переменных (`nextIndex`, `total`, массив воркеров);
  - одновременное количество активных промисов ограничено `limit`, что лучше, чем `O(n)` активных промисов при простом `Promise.all(tasks.map(fn => fn()))`.
